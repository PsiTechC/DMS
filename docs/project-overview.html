1 · What This Project Is

Every physical device in an organisation gets a printed QR sticker. Scan it with any phone and the device's full record opens instantly — specs, photos, videos, manuals, warranty, service history. If it's broken, raise a ticket in two taps and the admin has an email before you've walked away.

The problem it solves

Most organisations track assets in a spreadsheet that nobody updates, or in a system nobody opens because it lives on a desktop in another building. The information exists but is never at the device, which is exactly where it's needed — when a machine fails at 2am, or an auditor asks who holds asset AST-1042, or a warranty quietly lapses on 400 laptops at once.

This system puts the record on the device itself. The sticker is the index. No app to install, no login to browse, no training required — a phone camera is the entire client.

Who it's for

IT companies, factories, hospitals, educational institutions, and manufacturing organisations — anywhere physical assets are assigned to people, carry warranties, and break.

3 · The QR Lifecycle

This is the core workflow the entire system is built around:

1. GENERATE Admin creates a batch → DMS000001, DMS000002, DMS000003 … status: available │ 2. PRINT Download A4 label sheet PDF (28 per page, cut guides) │ 3. STICK Physically attach the labels to devices │ 4. SCAN Anyone points a phone at the sticker │ ▼ ┌──────────────────────────┐ │ Is this QR mapped yet? │ └──────────────────────────┘ │ ┌────────────┴────────────┐ │ NO │ YES ▼ ▼ "This QR is not DEVICE DETAILS PAGE assigned to any Overview · Specs · Images device. Please Videos · Manuals · Service login as Admin." Warranty │ │ ▼ ▼ Admin logs in "Raise a query" │ │ ▼ ▼ 5. MAP — fill form, Ticket saved → email to admin upload media, save DMS-2026-000001 │ ▼ status: mapped (one device per QR, unless remapped)
Why scanning needs no login The whole point of a sticker on a wall is that anyone can read it. Forcing a login before showing a device's manual would defeat the purpose. Raising a ticket does require login — that's how the reporter's identity gets attached automatically.

6 · Security

ControlImplementation
Password storagebcrypt hashed. Never returned by any API — the field is json:"-" at the model level, so it cannot leak by accident.
AuthenticationJWT HS256, 12-hour expiry, signed server-side.
AuthorisationRBAC enforced per route group. The role is re-read from the database on every request, so deactivating a user takes effect immediately rather than whenever their token happens to expire.
Rate limitingPer-IP token bucket. Login is throttled far harder (0.2 req/s, burst 8) than the global ceiling, to blunt credential stuffing.
File uploadsExtension and sniffed content-type must both match — a .png that is actually a script is rejected. Stored under a random UUID filename, never the client-supplied one.
SQL injectionParameterised queries throughout. ORDER BY columns are whitelisted, since they cannot be parameterised.
EnumerationUnknown email and wrong password return the identical message, so the login form cannot be used to discover which accounts exist.
Audit trailEvery mutating action logs user, role, action, reference, IP, and timestamp.
SecretsSMTP and DB credentials live in .env, which is gitignored and never sent to the browser.

7 · Questions & Answers

The questions people actually ask when they first see this system — grouped by who tends to ask them.

General & Business
What problem does this actually solve?

Asset information is normally stored far away from the asset. This puts it on the asset. Scan the sticker, get the record. It removes the "who owns this, when does its warranty end, where's the manual" scramble entirely — and it makes reporting a fault take seconds instead of an email thread.

Who can use it — do people need to install an app?

No app. Any phone camera opens the device page in a browser. Viewing needs no login at all; only raising a ticket does.

What kind of organisations is this for?

Anywhere physical assets are assigned to people and break: IT companies, factories, hospitals, schools, manufacturing. Anything with a serial number and a warranty fits the model.

How much does it cost to run?

No per-user or per-scan licensing — it's your own code on your own server. Email goes out over your existing SMTP account, so there's no per-email cost either. The real costs are a server, a database, and sticker paper.

QR Codes
How are QR codes generated, and are they guaranteed unique?

Yes, guaranteed. IDs come from a database counter incremented with a single atomic UPDATE … RETURNING statement. Two admins clicking "Generate 500" at the same instant get two separate, non-overlapping blocks — a collision is not merely unlikely, it's impossible by construction.

What's actually inside the QR code?

A URL: https://yourdomain.com/device/DMS000001. Nothing else — no data is embedded in the code itself, so the record can change freely without reprinting the sticker.

What happens the first time someone scans an unmapped sticker?

They see: "This QR is not assigned to any device. Please login as Admin." If an admin is already logged in, they instead get a direct button to map it. Either way the scan is still logged.

Can one QR code be mapped to two devices?

No. It's enforced at two levels: a check in the mapping handler, and a unique database constraint on qr_code_id. Even if two admins submit the mapping form simultaneously, the database rejects the second one.

What if a sticker is damaged, lost, or peels off?

Mark the QR as lost or replaced — scanning it then reports it's out of use rather than showing the device. Reprint a single large QR PDF for the same code, or unmap and remap to a fresh code.

How many QR codes can it handle?

5000 per batch, unlimited batches. The ID format holds 999,999 codes before it needs another digit. Inserts are batched 500 at a time, so generating 5000 is a handful of round trips, not 5000.

Can I change the domain after printing stickers?

This is the one genuinely painful mistake. The URL is baked into the QR image at generation time — it is not resolved when scanned. Change the domain and every already-printed sticker points at the old address. Set PUBLIC_BASE_URL to the real domain before generating production codes. Existing codes can be repointed in the database only if the domain still resolves.

Do I need a special printer or label stock?

Any printer. The sheet is plain A4 with 28 labels and cut guides — use adhesive sticker paper. Print at 100% scale; "fit to page" will shrink the codes and can make them unreadable.

Queries & Email
Where does a query go — the database or email?

Both, deliberately. The ticket is saved to the database first; the email is a notification, not the storage. If email were the only copy, one mail outage would lose tickets permanently, and there'd be no queries list, no status tracking, and no reports.

What if the mail server is slow or down — do we lose the ticket?

No. The ticket commits to the database, then the email is attempted on a background goroutine. A dead SMTP server produces a log line, not a failed submission. The user always gets their ticket number.

What's in the admin email?

Everything: ticket number, device number, QR number, device name, brand, model, serial, company, project, department, assigned employee, location, reporter name, employee ID, email, priority, title, description, and timestamp. Any attachment comes through attached. Reply-To is set to the reporter, so replying from the inbox reaches them directly.

How do I change who receives the notifications?

Change ADMIN_EMAIL in backend/.env and restart the backend. It's independent of SMTP_USERNAME (the account that sends) — the two are commonly confused. Note .env is read once at startup, so a restart is required.

Why doesn't the user type the device details on the query form?

Because they'd get them wrong, and because we already know them. The user enters only title, description, and priority. All 16 device and reporter fields are read server-side from the database — the form displays them so the user can see what's attached, but never sends them. This also means the client cannot forge which device a ticket is against.

Why does a ticket store copies of the device fields instead of just linking?

Deliberate denormalisation. A ticket is a historical record. If a laptop is reassigned from Rahul to Priya next month, a ticket Rahul raised must still say Rahul — otherwise history silently rewrites itself and audits become meaningless.

Are ticket numbers sequential? Can two collide?

Sequential per year — DMS-2026-000001. Same atomic counter as QR IDs, and the number is allocated inside the same transaction as the insert, so a failed insert never burns a number.

Does the reporter get told when their ticket is resolved?

Yes — every status change emails them, including the admin's remarks. That send is best-effort too: an undeliverable reporter address is logged and never blocks the admin's update.

Technical & Architecture
Why Go instead of Node or Python?

It compiles to one binary with no runtime to install or keep patched, starts instantly, and handles high concurrent scan traffic on small hardware. For a system whose load is thousands of short-lived scan requests, that fits well.

Why PostgreSQL rather than MySQL or MongoDB?

The core invariants are relational and need real transactions — one QR maps to exactly one device; a ticket number and its row must be allocated atomically. Postgres also provides COUNT(*) FILTER (WHERE …), which lets each dashboard card come from one query rather than many.

Is the API secure if someone bypasses the UI?

Yes — the UI is not a security boundary. It hides what a role can't do purely to avoid dead ends; every rule is enforced server-side per route group. Calling the API directly with a User token still gets a 403 on admin routes. This is tested per role.

What happens if I deactivate a user mid-session?

They lose access on their very next request. The role and active flag are re-read from the database on every call rather than trusted from the token — so you never have to wait out a token's 12-hour expiry.

What happens when a device is deleted?

The device, its media rows, and its service history are removed in one transaction, and its QR code returns to available for reuse. The physical files are deleted only after the transaction commits — so a rolled-back delete never leaves the database pointing at missing files. Tickets raised against it survive, because they hold their own snapshot.

Can uploaded files be malicious?

Both the extension and the sniffed content-type must match before anything is written — a script renamed .png is rejected. Files are stored under random UUID names, so a client-supplied filename can't traverse paths or overwrite anything.

Is there an audit trail?

Yes — QR generated, QR mapped, device created/updated/deleted, user login, QR scanned, query submitted, status changed, report exported. Each row records user, role, action, reference, IP, and timestamp, is filterable by date and action, and exports to Excel.

Does it work on mobile?

Fully responsive throughout — and mobile is the primary scanning surface, so the device page is designed phone-first.

What if the scanning device has no camera?

Three ways in: live camera, upload a photo or paste a screenshot (Ctrl+V) which decodes the image with no camera permission at all, or type the code by hand.

How does it scale?

Every list endpoint is paginated and capped at 200 rows per page. Filterable and searched columns are indexed. Report exports stream as a download rather than buffering in the browser. The realistic first bottleneck is video storage on disk, not the database.

Operations & Deployment
What do I need to run it?

PostgreSQL, and the compiled Go binary plus the built frontend. In development it's Go 1.26 and Node. Tables and demo accounts are created automatically on first run — there's no separate migration step.

How do I back it up?

Two things: pg_dump the database, and copy the uploads/ directory. The database holds every record; uploads/ holds the images, videos, and manuals. Backing up only one gives you a broken restore.

What must change before going live?
  1. Set a long random JWT_SECRET (64+ chars).
  2. Change all three seeded demo passwords.
  3. Set APP_ENV=production.
  4. Point PUBLIC_BASE_URL at the real domain before generating any QR codes.
  5. Use a dedicated database role, not postgres.
  6. Put the API behind HTTPS.
  7. Add auth to /uploads if manuals are confidential.
Can multiple clients be isolated from each other?

Not yet — today a Client sees every ticket and device. If you onboard multiple external clients who must not see each other's data, scope queries and devices by the client's company field. Worth doing before real client accounts exist, not after.

What about WhatsApp notifications?

Built and wired for the Meta Cloud API, currently disabled. The message builder is done. Set WHATSAPP_ENABLED=true plus the token and phone ID to switch it on — no code change needed.

What are the known limitations?
  • No multi-tenant isolation between clients (see above).
  • No password-reset email flow — an admin resets passwords.
  • Uploads go to local disk, not S3 — fine on one server, needs shared storage to run several.
  • WhatsApp untested against a live account.
  • No offline scanning; the device page needs a connection.

9 · Notable Design Decisions

Five choices worth understanding, because each one has a non-obvious reason:

1 · Atomic counters, not MAX(id)+1

Asset IDs and ticket numbers come from a single UPDATE counters SET value = value + n … RETURNING. The naive read-then-write approach silently produces duplicates the moment two admins act at once — and that is exactly the kind of bug that only shows up in production.

2 · Tickets snapshot their device

A query copies the device and reporter fields at submit time rather than joining. It costs storage but keeps history truthful: reassigning a device later must not rewrite who reported what.

3 · Notifications never block the request

Email is sent on a background goroutine after the ticket commits. A slow SMTP server must never make a user think their query failed.

4 · Public scan, authenticated action

Reading a device needs no login — that's the point of a sticker. Raising a ticket does, which is how the reporter's identity is attached automatically and cannot be forged.

5 · Files deleted after commit, never before

Deleting a device removes its rows in a transaction, and only touches the disk once that transaction has committed. Deleting files first would leave the database pointing at nothing if the transaction rolled back.
